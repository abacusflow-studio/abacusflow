# AbacusFlow Mobile 录入优先整改方案

## 1. 背景与目标

当前 `abacusflow-apps/apps/mobile` 已经从模板应用推进到可登录、可连后端、可创建基础单据的状态，并且已经开始加入扫码能力。但它的产品气质仍然混合了两种方向：

- 一部分页面在做移动录入：`app/(tabs)/index.tsx` 的“扫码录入”、`app/scan/index.tsx`、采购/销售新增页里的扫码添加。
- 另一部分页面仍像“管理后台移动版”：产品列表、库存列表、订单列表、客户/供应商/用户管理、详情和编辑入口都占了很高权重。

本次整改的核心不是继续把 Web 管理后台搬到手机里，而是把 mobile 明确定位为：

> 门店/仓库一线人员使用的快速录入工具。优先完成扫码、入库、出库、新品建档、草稿恢复和提交反馈；查询和管理只服务于录入过程，不作为主目标。

## 2. 产品原则

1. **录入优先**
   - 打开 app 后第一屏应该直接能开始“扫码入库 / 扫码出库 / 新品建档”。
   - 管理类页面不应占主导航核心位置。

2. **少输入，快确认**
   - 默认值要主动给：今天日期、最近供应商/客户、常用仓库、默认数量 1、默认折扣 1。
   - 非必填字段收进“更多信息”，不要打断主流程。

3. **扫码是主路径，手动选择是兜底**
   - 扫码失败、没有相机权限、条码不存在时，才进入手动输入/选择。
   - 不要让用户先进入复杂列表再点“新增”。

4. **失败可恢复**
   - 录到一半退出、网络失败、接口报错，都不能丢数据。
   - 提交失败应保留草稿，并给出“重试 / 稍后提交 / 修改后再提交”。

5. **查询服务录入**
   - 库存、产品、订单列表保留，但应变成“确认数据是否录对、查找最近记录、从查询结果继续录入”的辅助功能。

## 3. 现状问题清单

### 3.1 信息架构仍偏管理

当前底部 Tab：

- 首页
- 库存
- 记录
- 更多

问题：

- “库存”“记录”“更多”更像管理端入口，不像录入端任务入口。
- `more.tsx` 里大量使用“管理”语言：数据管理、客户管理、供应商管理、储存点管理、用户管理。
- 用户管理这类后台能力不应在一线录入 app 的主路径中出现。

整改方向：

- 将 Tab 改成任务导向。
- 将管理功能降级到隐藏/二级入口，或提示“请到 Web 端管理”。

建议 Tab：

| Tab | 目标 | 内容 |
| --- | --- | --- |
| 录入 | 默认首页 | 扫码入库、扫码出库、新品建档、盘点/调整入口 |
| 草稿 | 失败恢复 | 未提交草稿、提交失败队列、最近录入 |
| 查询 | 录入辅助 | 扫码查库存、产品/库存/订单搜索 |
| 我的 | 配置与账户 | 当前用户、后端环境、登出、版本、相机权限状态 |

### 3.2 首页已经有录入雏形，但还混杂概览和管理

当前 `app/(tabs)/index.tsx` 有：

- 大按钮“扫码录入”
- 快捷操作：扫码入库、扫码出库、新增产品、产品列表
- 概览统计：产品、库存、低库存

问题：

- “产品列表”是管理入口，放在快捷操作里会稀释录入目标。
- 概览统计适合老板/管理者，不是店员录单的首要任务。
- “扫码入库”和“扫码出库”都跳到同一个 `/scan`，扫码后才选择动作，任务感不够明确。

整改方向：

- 首页改为“录入工作台”。
- 顶部只保留非常轻的状态信息：当前用户、网络/后端状态、今日已提交数。
- 主区域放三个大操作：
  - 扫码入库
  - 扫码出库
  - 新品建档
- 次级操作放小入口：
  - 手动入库
  - 手动出库
  - 扫码查库存
  - 继续草稿

### 3.3 采购/销售录入仍是表单页，不是录入流

当前 `order/purchase/add.tsx` 和 `order/sale/add.tsx`：

- 页面加载供应商/客户、产品、库存单元。
- 用横向 chip 选择供应商/客户。
- 明细区每项都有数量、单价输入。
- 手动选择通过 `Alert.alert` 展示最多 10 条。

问题：

- 数据多时 `Alert.alert` 无法搜索、无法分页、无法展示库存/价格等上下文。
- 手机小屏上字段一次性铺开，录入负担较重。
- 供应商/客户选择应支持“最近使用 / 默认散客 / 快速新增”，而不是只能从已有 chip 中选。
- 提交失败后只有弹窗，没有草稿/重试队列。

整改方向：

- 把“新增采购单/销售单”改成分阶段录入流。
- 使用底部吸附操作栏，始终显示当前明细数、合计、提交按钮。
- 将选择器改为可搜索 Bottom Sheet，不再用 `Alert.alert` 做数据选择。

建议采购入库流程：

1. 选择供应商
   - 默认展示最近 3 个供应商。
   - 支持搜索。
   - 支持“快速新增供应商”，只填名称和电话。
2. 扫描/添加产品
   - 扫到已存在产品：加入明细，数量默认为 1，焦点进入数量。
   - 扫到不存在条码：进入新品建档，保存后回到当前采购单并自动加入明细。
   - 手动添加：打开可搜索产品选择器。
3. 补数量和单价
   - 数量用步进器 + 数字输入。
   - 单价可为空时应明确业务规则；若后端必填，默认 0 或要求录入要在 UI 上说清楚。
4. 复核提交
   - 展示供应商、产品数、合计、备注。
   - 成功后显示“继续录下一单 / 查看单据 / 回到录入”。

建议销售出库流程：

1. 选择客户
   - 支持“散客/默认客户”作为快捷项，如果业务允许。
   - 支持快速新增客户。
2. 扫描商品条码
   - 物料类：展示可用库存和库位，默认数量 1。
   - 资产类：必须扫描或选择 SN，不能只按产品条码出库。
3. 校验库存
   - 数量不可超过可用库存。
   - 库存不足要展示“当前可用数量”和恢复动作。
4. 复核提交
   - 展示客户、明细数、合计、折扣。
   - 折扣默认隐藏在“更多”，除非业务强依赖。

### 3.4 销售扫码存在数据匹配风险

当前销售扫码逻辑里，`SelectableInventoryUnit` 没有足够的产品身份字段，代码只能从所有可用库存单元中找第一个未选项，或用 `title.includes(sn)` 匹配 SN。

风险：

- 扫 A 产品条码，可能加入 B 产品库存单元。
- SN 用标题字符串匹配，不稳定。
- 对资产类商品尤其危险，可能造成错误出库。

整改要求：

- 后端或 OpenAPI DTO 应补充库存单元的产品身份字段：
  - `productId`
  - `productName`
  - `barcode`
  - `serialNumber`，如果资产有独立 SN
  - `depotId/depotName`
  - `availableQuantity`
- mobile 侧销售扫码必须按 `productId/barcode/serialNumber` 精确匹配。
- 在 DTO 未补齐前，销售扫码只能作为受限功能：
  - 允许扫码后跳到搜索结果页让用户确认。
  - 不允许静默加入第一条库存单元。

### 3.5 新品建档应服务入库，而不是独立产品管理

当前 `product/add.tsx` 已支持扫描条码，字段包括：

- 条码
- 产品名称
- 规格
- 产品类型
- 单位
- 类别
- 备注

问题：

- 对现场录入来说，字段仍偏多。
- 类别必填，如果类别列表为空或店员不知道类别，会卡住录入。
- 创建成功后只是 `router.back()`，没有“保存并入库”的连续动作。

整改方向：

- 新品建档分成“必填”和“稍后补充”。
- 必填建议：
  - 条码
  - 产品名称
  - 类型：物料 / 资产
  - 单位
- 可稍后补充：
  - 类别
  - 规格
  - 备注
  - 安全库存
- 当从采购入库流程进入新品建档时，保存后必须回到当前采购单并自动加入该产品。
- 提供两个提交动作：
  - 保存
  - 保存并入库

### 3.6 缺少草稿、失败队列和离线降级

录入 app 的高频场景是移动网络不稳定、扫码中断、突然切后台。

必须补齐：

- 本地草稿：采购单、销售单、新品建档。
- 自动保存：关键字段变化后 debounce 保存。
- 失败队列：接口失败时保留请求体和错误原因。
- 草稿列表：Tab 里单独展示。
- 明确状态：
  - 未完成
  - 待提交
  - 提交失败
  - 已提交

技术建议：

- 第一版可以使用 AsyncStorage 或 SecureStore 之外的普通本地存储方案。令牌仍走 `lib/secure-storage.ts`。
- 草稿 key 建议：
  - `abacusflow.mobile.draft.purchase`
  - `abacusflow.mobile.draft.sale`
  - `abacusflow.mobile.draft.product`
  - `abacusflow.mobile.submitQueue`
- 每个草稿存：
  - `id`
  - `type`
  - `updatedAt`
  - `payload`
  - `summary`
  - `lastError`

## 4. 目标信息架构

### 4.1 路由建议

保留 Expo Router，但调整语义：

```txt
app/
  (tabs)/
    index.tsx              # 录入工作台
    drafts.tsx             # 草稿和失败队列
    lookup.tsx             # 查询
    me.tsx                 # 我的
  entry/
    scan.tsx               # 通用扫码，支持 mode 参数
    purchase.tsx           # 采购入库录入流
    sale.tsx               # 销售出库录入流
    product.tsx            # 新品建档录入流
  lookup/
    inventory.tsx
    product.tsx
    order.tsx
  settings/
    account.tsx
```

如果短期不想大迁移，也可以先保留现有路由，做软调整：

- `/(tabs)/index` 改为“录入”
- `/(tabs)/inventory` 改名为“查询”
- `/(tabs)/orders` 改名为“流水”
- `/(tabs)/more` 改名为“我的”
- `products/depots/user` 仍隐藏，不进入主 tab。

### 4.2 首页结构

建议首屏结构：

1. 顶部状态条
   - 左侧：当前门店/用户
   - 右侧：在线/离线、同步失败数
2. 主按钮
   - “扫码入库”：占宽 100%，高度 88-104
   - “扫码出库”：占宽 100%，高度 88-104
3. 次级操作网格
   - 新品建档
   - 手动入库
   - 手动出库
   - 扫码查库存
4. 草稿提示
   - 如果有未完成草稿，显示一条紧凑横幅：“有 2 条未提交，继续处理”
5. 今日记录
   - 只展示最近 3 条提交成功记录，不展示完整统计面板。

## 5. 视觉与基础格调

### 5.1 关键词

目标气质：

- 可靠
- 快速
- 清楚
- 门店/仓库可用
- 少装饰

避免：

- 后台管理感过强
- 过多彩色卡片
- 大面积阴影
- 报表/KPI 风格
- “系统管理”式文案

### 5.2 色彩建议

当前颜色主要来自 `@abacusflow/ui-tokens`：

- 主色：`#1677ff`
- 背景：`#f5f5f5`
- 成功：`#52c41a`
- 警告：`#fa8c16`
- 危险：`#ff4d4f`

第一版可以不推翻 token，但要收敛使用：

- 主色只用于主操作、选中态、扫码框。
- 入库不必固定橙色，出库不必固定紫色；建议使用主色 + 语义标签，减少彩色碎片。
- 危险色只用于删除、库存不足、提交失败。
- 背景建议从 `#f5f5f5` 调整为更干净的 `#F7F8FA`。
- 卡片边框优先于阴影，弱化“浮层感”。

建议新增语义 token：

```ts
entryPrimary: "#155EEF"
entryPrimarySoft: "#EAF1FF"
surface: "#FFFFFF"
surfaceMuted: "#F7F8FA"
textStrong: "#1F2937"
textMuted: "#6B7280"
borderSoft: "#E5E7EB"
scanFrame: "#155EEF"
successSurface: "#ECFDF3"
warningSurface: "#FFF7E8"
dangerSurface: "#FEF3F2"
```

### 5.3 字体与密度

- 标题：18-22，少用 28 以上的大标题。
- 表单 label：13-14，正文输入：16，保证现场可读。
- 明细卡商品名：15-16，数量/金额：18-22。
- 不使用负字距。
- 不按屏幕宽度缩放字体。

### 5.4 触控规范

- 所有可点区域最小 44x44。
- 相邻按钮至少 8px 间距。
- 底部主按钮高度 52-56。
- 扫码页关闭按钮、重新扫描按钮必须可单手点击。
- 删除明细不要只给 20px 图标，应给 44px 命中区。

### 5.5 文案规范

把“管理”换成“录入/查询/设置”语言：

| 旧文案 | 新文案 |
| --- | --- |
| 首页 | 录入 |
| 记录 | 流水 |
| 更多 | 我的 |
| 产品列表 | 查产品 |
| 客户管理 | 客户资料 |
| 供应商管理 | 供应商资料 |
| 用户管理 | 账号与权限，且建议只在 Web 管理 |
| 新增采购单 | 入库 |
| 新增销售单 | 出库 |
| 创建采购单 | 提交入库 |
| 创建销售单 | 提交出库 |

## 6. 组件整改建议

### 6.1 新增录入专用组件

建议在 `abacusflow-apps/packages/ui-native/src` 或 mobile 本地先建立：

1. `EntryActionCard`
   - 用于首页主动作。
   - 支持 icon、title、description、statusBadge。

2. `EntryBottomBar`
   - 固定底部，显示明细数、合计、保存草稿、提交。

3. `EntryItemCard`
   - 订单明细卡。
   - 内置数量步进器、单价输入、删除命中区。

4. `SearchSelectSheet`
   - 替代 `Alert.alert` 选择。
   - 支持搜索、分页、空状态、最近使用。

5. `ScanSheet` / `ScanResultPanel`
   - 扫码完成后的结果确认。
   - 不要每次扫码后跳多层路由，尽量在当前流中处理。

6. `DraftBanner`
   - 首页提示未完成录入。

7. `SubmitState`
   - 提交中、成功、失败、重试。

### 6.2 调整共享组件

`ListScreen`：

- 保留给查询页。
- 不再作为主业务入口。
- 增加 empty action，例如“扫码新增”。

`FormScreen`：

- 保留给低频资料维护。
- 不再用于采购/销售主流程。
- 字段错误要从 Alert 改为 inline error。

`BarcodeScanner`：

- 增加 torch 手电筒开关，如果 expo-camera 支持当前平台。
- 增加扫描冷却和重复扫码提示。
- 增加 `mode` 文案：入库扫码、出库扫码、查库存扫码、扫 SN。
- 权限失败页要给“去系统设置”的指引。

## 7. 技术整改任务拆分

### P0：先堵住录入安全风险

1. 修销售扫码匹配
   - 后端 DTO 补齐库存单元与产品/SN 字段。
   - mobile 禁止用第一条可用库存单元兜底。
   - 没有精确匹配时必须让用户确认。

2. 替换 `Alert.alert` 手动选择
   - 采购产品选择改为搜索 sheet。
   - 销售库存单元选择改为搜索 sheet。
   - 支持超过 10 条。

3. 增加草稿保存
   - 采购、销售、新品建档都要自动保存草稿。
   - 提交成功后清理草稿。
   - 提交失败后保留草稿和错误。

4. 确保 SecureStore 只通过包装层使用
   - auth 代码应继续 `import * as SecureStore from "@/lib/secure-storage"`。
   - 不要在业务代码中直接 import `expo-secure-store`。

### P1：重塑主导航和录入流

1. Tab 改名和重排
   - 首页 -> 录入
   - 库存 -> 查询
   - 记录 -> 流水或草稿，按实际实现选择
   - 更多 -> 我的

2. 首页重做
   - 主动作只放录入任务。
   - 管理入口移到“我的/设置/资料”二级区域。

3. 采购/销售录入改为流式布局
   - 顶部是任务标题和当前步骤。
   - 中部是当前明细。
   - 底部固定操作栏。

4. 新品建档改为最小必填
   - 支持从扫码结果带条码进入。
   - 支持保存后返回上游流程。

### P2：视觉系统落地

1. 收敛颜色
   - 扫码、提交、选中态用统一主色。
   - 删除、失败、库存不足用危险色。
   - 入库/出库不再靠大面积橙色/紫色区分。

2. 减少卡片阴影
   - 主容器用白底 + 细边框。
   - 只在 modal/sheet 上使用阴影。

3. 统一控件尺寸
   - 输入框高度 48。
   - 主按钮高度 52-56。
   - 图标按钮命中区 44。

4. 统一反馈
   - 保存成功：轻提示 + 明确下一步。
   - 失败：错误面板 + 重试按钮。
   - 加载：骨架或行内 loading，不要全局空白。

### P3：发布前质量

1. 真机验证
   - Android development build。
   - iOS development build，如果计划 iOS。
   - 相机权限、扫码、后台切换、弱网。

2. 回归场景
   - 新品扫码建档后入库。
   - 已有产品扫码入库。
   - 物料扫码出库。
   - 资产扫码 SN 出库。
   - 网络失败后恢复草稿并重试。

3. 自动化
   - 至少补核心纯逻辑单测：草稿序列化、表单校验、扫码匹配。
   - 组件测试覆盖 `SearchSelectSheet`、`EntryBottomBar`。

## 8. 开发落地顺序

建议开发者按下面顺序做，不要先大规模改样式：

1. **修销售扫码的数据正确性**
   - 这是最高风险。
   - 未完成前，不要发布销售扫码出库。

2. **做草稿和失败恢复**
   - 没有草稿，录入 app 的可信度不够。

3. **替换 Alert 选择器**
   - 这会明显提升采购/销售录入效率。

4. **重做首页和 Tab 文案**
   - 让用户一打开就知道这是录入工具。

5. **重构采购/销售页面布局**
   - 从表单页转为录入流。

6. **收敛视觉 token**
   - 统一风格，减少后台感。

7. **真机/EAS 验证**
   - 尤其是 `expo-camera`，新增原生模块后不能只依赖 Expo Go。

## 9. 验收标准

### 9.1 产品验收

- 新用户打开 app，3 秒内能找到“扫码入库”和“扫码出库”。
- 完成一次扫码入库，不需要进入产品列表或订单列表。
- 扫到不存在条码，可以建档并回到原入库流程。
- 销售出库不会把错误库存单元加入明细。
- 录到一半退出 app，再回来可以继续。
- 网络失败后不会丢单。
- “用户管理/系统管理”等后台语言不出现在主流程。

### 9.2 UI 验收

- 主要点击区域不小于 44x44。
- 底部提交按钮在采购/销售录入页始终可见。
- 错误不只通过弹窗表达，页面内也能看到恢复动作。
- 页面颜色不再是多种彩色卡片混用。
- 在 375px 宽度设备上，按钮文字不溢出、不重叠。

### 9.3 技术验收

- `npm run typecheck -w abacusflow-mobile` 通过。
- `npm run lint -w abacusflow-mobile` 通过。
- Android development build 能打开相机并扫码。
- 采购/销售/新品建档主流程在真机上跑通。
- 关闭网络提交失败后，草稿仍能恢复并重试。

## 10. 需要后端配合的点

1. 库存单元选择接口增强
   - 补 `productId/barcode/serialNumber/depotName/availableQuantity`。

2. 扫码查询接口
   - 建议提供按 barcode 查询产品的接口，避免 mobile 每次拉全量 selectable products 再前端过滤。

3. 最近使用数据
   - 供应商、客户、仓库可以按用户返回最近使用项。

4. 草稿是否服务端化
   - 第一版可本地草稿。
   - 后续如果多设备录入，需要服务端草稿。

## 11. 不建议现在做的事

- 不要把 Web 的所有管理能力继续搬到 mobile。
- 不要先做报表/KPI/复杂统计。
- 不要让销售扫码在 DTO 不完整时直接创建出库明细。
- 不要继续用弹窗承载复杂选择。
- 不要只改颜色和卡片样式而不改录入流程。
